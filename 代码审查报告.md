## 二、安全问题

### 2.1 【严重】管理接口无鉴权，可被任意访问

**位置**: `backend/server.js`  
- `GET /api/inquiries`：返回全部留言，未校验身份  
- `PATCH /api/inquiries/:id`：任何人可回复/标记留言  
- `GET /api/reservations`：可按日期或手机号查询全部预约，未鉴权  

**风险**: 用户隐私泄露；恶意用户可篡改/删除留言与预约数据。

**修改方案**:

1. 为管理端接口增加鉴权中间件（如 JWT 或 Session），仅登录管理员可访问：
   - 将 `GET /api/inquiries`（无 phone 参数时）、`PATCH /api/inquiries/:id`、`GET /api/reservations` 等视为管理接口，要求请求头携带有效 token。
2. 或单独增加管理路由前缀（如 `/admin/api/inquiries`），并对 `/admin/*` 统一做鉴权。

示例（中间件思路）：

```javascript
// 鉴权中间件示例：从 Authorization: Bearer <token> 或 Cookie 校验登录态
function requireAuth(req, res, next) {
  const token = req.headers.authorization?.replace('Bearer ', '') || req.cookies?.auth_token;
  if (!token || !validateToken(token)) {
    return res.status(401).json({ success: false, message: '请先登录' });
  }
  req.user = getUserIdFromToken(token);
  next();
}
// 对管理接口应用：app.get('/api/inquiries', requireAuth, handler);
// 注意：GET /api/inquiries?phone=xxx 可保留为公开，供用户凭手机号自查留言
```

### 2.2 【严重】生产环境错误信息泄露

**位置**: `backend/server.js` 多处 `catch` 中返回 `error: err.message`。

例如：

- 第 54–56 行：`/login` 返回 `err.message`  
- 第 96、127、153、184、268、291、371 行：多个接口在 500 时返回 `error: err.message`

**风险**: 暴露数据库结构、路径、配置等内部信息，便于攻击者探测与利用。

**修改方案**:

- 生产环境统一不向客户端返回 `err.message`，仅返回通用提示（如「服务器错误」）。
- 将详细错误仅写入日志（如 `console.error` 或日志库），便于排查。

示例：

```javascript
const isDev = process.env.NODE_ENV !== 'production';
// 在 catch 中：
console.error('Login error:', err.message);
return res.status(500).json({
  success: false,
  message: '服务器错误',
  ...(isDev && { error: err.message })
});
```

### 2.3 【严重】CORS 配置为 `*`，生产环境不安全

**位置**: `backend/server.js` 第 13–17 行。

```javascript
res.setHeader('Access-Control-Allow-Origin', '*');
```

**风险**: 任意网站可向该后端发起跨域请求，若结合 Cookie/登录态会放大 CSRF 与数据泄露风险。

**修改方案**:

- 生产环境改为白名单：只允许前端所在域名（如 `https://your-domain.com`）。
- 开发环境可继续使用 `*` 或 `http://localhost:*`。

示例：

```javascript
const allowedOrigins = process.env.NODE_ENV === 'production'
  ? (process.env.ALLOWED_ORIGINS || 'https://your-domain.com').split(',')
  : ['http://localhost:3000', 'http://127.0.0.1:5500', 'null'];
const origin = req.headers.origin;
if (allowedOrigins.includes(origin)) {
  res.setHeader('Access-Control-Allow-Origin', origin);
}
```

### 2.4 【严重】前端登录为模拟逻辑，未对接后端且存在硬编码凭证

**位置**: `frontend/js/login.js`  
- 第 352–401 行：`login()` 使用 `setTimeout` 模拟，未请求后端 `POST /login`。  
- 第 361 行：硬编码 `admin` / `123456`；第 381 行：短信验证码硬编码 `123456`。

**风险**:  
- 真实后端登录接口未被使用，部署后无法真正登录。  
- 前端写死账号与验证码，易被滥用且误导维护人员。

**修改方案**:

1. 账号密码登录改为调用后端：  
   - 使用 `window.API_BASE`（或与 `config.js` 一致）拼出 `POST ${API_BASE}/login`，提交 `{ username, password }`，根据返回的 `success` 与 `message` 处理成功/失败与验证码刷新。
2. 删除前端所有硬编码的 `admin`/`123456` 及固定短信验证码；短信验证码必须由后端发送并校验。
3. 登录成功后，若后端返回 token 或用户信息，再写入 `localStorage`（如 `auth_token`、`user_info`），与现有 `main.js` 的 `checkLoginStatus` 保持一致。

### 2.5 【中等】init-db.sql 中明文密码

**位置**: `backend/init-db.sql` 第 15 行。

```sql
INSERT INTO testdb.users (username, password) VALUES ('admin', '123456');
```

**风险**: 若仅执行 `init-db.sql` 而未执行 `npm run seed`，数据库中会存在明文密码。

**修改方案**:

- 在 `init-db.sql` 中删除该 INSERT，或在文件顶部注释说明：**必须先执行 `npm run seed` 写入 bcrypt 哈希，不要直接使用本文件中的用户数据**。
- 推荐：`init-db.sql` 只建库、建表，用户数据一律由 `seed.js` 插入（已用 bcrypt）。

---

### 2.6 【中等】登录接口存在用户枚举

**位置**: `backend/server.js` 第 33–50 行。

当前逻辑：用户不存在与密码错误均返回 `{ success: false }`，但控制台会打印「登录」成功/失败，响应体未区分「用户不存在」与「密码错误」。若将来在错误信息上区分，会存在用户枚举。

**修改方案**:

- 保持接口层统一返回「用户名或密码错误」，不区分具体原因。
- 避免在响应或日志中暴露「用户不存在」与「密码错误」的差异（当前实现已基本满足，后续扩展时注意保持）。

---

### 2.7 【中等】请求体大小未限制，易被滥用

**位置**: `backend/server.js` 第 10 行 `app.use(express.json())` 未配置 `limit`。

**风险**: 超大 JSON 请求体可能导致内存与解析时间问题，易被 DoS 利用。

**修改方案**:

```javascript
app.use(express.json({ limit: '256kb' }));
```

可根据实际接口需求调整 `limit`（如 64kb–512kb）。

---

### 2.8 【建议】敏感响应头与安全头

**建议**: 增加安全相关响应头，降低 XSS、点击劫持等风险。

示例（在全局或静态资源前）：

```javascript
app.use((req, res, next) => {
  res.setHeader('X-Content-Type-Options', 'nosniff');
  res.setHeader('X-Frame-Options', 'DENY');
  next();
});
```

## 三、代码质量问题

### 3.1 【严重】前端使用 innerHTML 拼接用户数据，存在 XSS 风险

**位置**: `frontend/js/main.js` 第 16 行。

```javascript
loginBtn.innerHTML = '<svg ...><span>' + user.name + '</span>';
```

**风险**: 若 `user.name` 来自不可信来源（如将来从接口或 localStorage 解析得到用户可控数据），未转义即插入 HTML 会导致 XSS。

**修改方案**:

- 使用 `textContent` 或先转义再插入。例如只更新文本部分，不整段写 `innerHTML`：

```javascript
var span = loginBtn.querySelector('span') || document.createElement('span');
span.textContent = user.name;
if (!loginBtn.contains(span)) loginBtn.appendChild(span);
```

- 或封装 `escapeHtml`，对 `user.name` 转义后再拼入 HTML（与 `inquiry-manage.html` 中用法一致）。

### 3.2 【中等】前端 API 基地址不统一

**位置**:  
- `frontend/js/config.js`：`window.API_BASE = 'http://localhost:3000'`  
- `frontend/js/login.js` 第 12 行：`CONFIG.API_BASE = '/api'`，且登录未调用后端。

**问题**: 配置分散，登录页未使用统一 base，也未对接真实接口。

**修改方案**:

- 登录请求使用 `window.API_BASE`（与 config.js 一致），调用 `POST ${window.API_BASE}/login`。
- 删除 login.js 中独立的 `CONFIG.API_BASE = '/api'`，或改为 `CONFIG.API_BASE = window.API_BASE || 'http://localhost:3000'`，保证与其它页面一致。

---

### 3.3 【中等】重复的错误处理与 SQL 片段拼接

**位置**: `backend/server.js` 多处（如 91–97、114–127、356–372 行）。

**问题**:  
- 多个路由中 `ER_NO_SUCH_TABLE` 与通用 500 的处理逻辑重复。  
- PATCH inquiries 中通过 `updates.join(', ')` 动态拼 SQL，虽参数化无误，但可读性差、后续扩展易出错。

**修改方案**:

- 抽一个统一错误处理中间件或工具函数，根据 `err.code === 'ER_NO_SUCH_TABLE'` 返回固定提示，其它 500 统一返回通用信息并写日志。
- PATCH 中可固定列（如 `admin_reply`、`replied_at`、`status`），用固定 SQL + 参数数组，避免动态拼接列名。

---

### 3.4 【建议】输入校验与限流

- **手机号/电话**: 后端对 `contact_phone`、`phone` 等做格式与长度校验（如正则、最大长度），避免无效或恶意数据入库。  
- **限流**: 对 `/login`、`/api/inquiries`、`/api/reservations` 等接口做按 IP 或用户的限流，防止暴力尝试与滥用。

## 四、性能问题

### 4.1 【中等】新闻列表重复查询总数

**位置**: `backend/server.js` 第 200–224 行。

当前逻辑：先查一页列表，再单独 `SELECT COUNT(*) FROM news`。在数据量较大时，两次查询可合并或优化。

**修改方案**:

- 若分页只需「是否有下一页」，可用 `limit + 1` 多查一条判断。  
- 若必须返回 `total`，可保留两次查询，但为 `COUNT(*)` 加索引（如 `news(sort_order, id)`），或考虑缓存总数（如短期 Redis/内存缓存），减少频繁全表 COUNT。

---

### 4.2 【建议】连接池与长连接

**位置**: `backend/server.js` 第 21–30 行。

当前 `connectionLimit: 10`、`queueLimit: 0` 合理。建议：  
- 正式环境通过环境变量配置 `connectionLimit` 与 MySQL `wait_timeout`，避免连接堆积。  
- 如有健康检查频繁调用，注意连接复用，避免短连接过多。

## 六、总结

- **安全**: 最优先处理管理接口鉴权、生产环境错误信息、CORS 以及前端登录对接与去除硬编码，其次处理明文密码、body 限制与安全头。  
- **代码质量**: 优先消除 main.js 中基于 `user.name` 的 XSS 风险，统一 API 基地址与登录流程，再考虑错误处理与 SQL 结构优化。  
- **性能**: 当前规模下问题不大，可针对新闻列表与连接池做小幅优化与配置化。
